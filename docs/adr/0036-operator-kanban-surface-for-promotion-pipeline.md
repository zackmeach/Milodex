# ADR 0036 — Operator Kanban surface for the promotion pipeline

**Status:** Accepted - visual spec only (canonical reference superseded; see Decision 1) · 2026-05-08
**Related:** [ADR 0033](0033-gui-runtime-is-pyside6-qt-quick.md) (GUI runtime), [ADR 0034](0034-phase-5-scope-orders-observability-before-features.md) (Phase 5 observability-first), [ADR 0035](0035-design-system-and-theme-architecture.md) (design system architecture), [ADR 0005](0005-kill-switch-manual-reset.md) (manual-reset semantic, applies to demotion governance), [PHASE5_PLANNING.md](../PHASE5_PLANNING.md), [STRATEGY_BANK.md](../STRATEGY_BANK.md), [DESIGN_SYSTEM.md](../DESIGN_SYSTEM.md), [`docs/prototypes/strategy-kanban-v0.3/`](../prototypes/strategy-kanban-v0.3/) (original visual reference; superseded by docs/mockups/), [`docs/mockups/bench-brief.md`](../mockups/bench-brief.md) (canonical visual reference), [`docs/mockups/bench-surface.html`](../mockups/bench-surface.html) (canonical visual reference)

## Context

[ADR 0034](0034-phase-5-scope-orders-observability-before-features.md) bounded Phase 5 to observability surfaces. By the end of Phase 5 the Anchor view (operator first-frame, kill-switch + market clock + account + positions) and the Strategy Bank surface (read-only render of paper-stage and blocked-at-backtest strategies) had shipped, plus the design-system foundation (tokens, three themes, components, editorial-press direction) and the QML-load smoke test. [ADR 0038](0038-phase-5-is-closed-and-phase-6-may-open.md) later closed Phase 5 after the distribution and installer source work landed. No feature surface — anything that gives the operator new strategy operations beyond the existing CLI — was in Phase 5 scope.

Late in Phase 5 the operator surfaced an interaction-language idea: drag-to-reorder rows in the Strategy Bank surface, possibly serializing deployment as a queue. Investigation (the drag-vibes prototype at [`docs/prototypes/strategy-bank-drag-vibes.html`](../prototypes/strategy-bank-drag-vibes.html), three iterations of variant trade-offs) validated a specific drag mechanic — variant B with the sticky 50% rule — that feels right.

The conversation then expanded substantially. The operator described a full Kanban-style operator surface for the promotion pipeline:

> "I would like in front of me to be every single strategy that I have access to. I would like to be able to start with all 12 or 13 runners or configs in the pre-backtest or untested category. Drag a few that I want into the backtest section. Hit start, which would then run backtesting on all the strategies in that section. Depending on the results, then update the small chip that's visible for that each strategy that shows whether this is backtested able to promote or backtested not able to be promoted. Then I can drag the few that are able to be backtested that have passed backtesting over to paper testing. I could even try and drag some of the items that are backtested not ready to promote into the paper testing, and then would be shown a denied message…There should also be on that strategies row a running or idle chip…"

The operator then iterated visually on the idea, producing a self-contained React prototype (v0.2 then v0.3, latest at [`docs/prototypes/strategy-kanban-v0.3/`](../prototypes/strategy-kanban-v0.3/)) that spans five columns (IDLE / BACKTEST / PAPER / MICRO LIVE / LIVE), full editorial-press treatment (roman-numeral column ticks, parchment dot-grain background, per-stage hue identity, italic Newsreader decks throughout), per-card promotion bar visualization, bulk-action buttons with smart-skip ×N counters, and the drag mechanic from the earlier prototype extended to cross-column promotions and demotions.

The operator's framing of the prototype, verbatim:

> "the mechanics of what is going on, how strategies are run, how the content in each card gets updated, doesn't have a whole lot of thought…I love the way it feels, the way it looks, and how it is laid out."

That framing splits the artifact cleanly into a layer that is locked (visual spec, layout, UX direction) and a layer that is not yet defined (mechanics, data model, runtime). This ADR adopts the visual spec and explicitly defers the mechanics layer to follow-up ADRs. The intent is to capture the surface's shape now — while the operator's thinking is fresh, before Phase 5 closes out — without committing to mechanics decisions that haven't been thought through yet.

The Kanban surface is **Phase 6+ implementation work**. Phase 5 close-out is not blocked on it. This ADR is a forward-looking design freeze, not a Phase 5 deliverable.

## Decision

1. **Adopt the bench mockup as the canonical visual reference for the Bench surface.** [`docs/mockups/bench-brief.md`](../mockups/bench-brief.md) and [`docs/mockups/bench-surface.html`](../mockups/bench-surface.html) are the reference artifacts. Future Phase 6 implementation PRs treat these as the visual contract: stage sections stacked top-to-bottom (see [ADR 0048](0048-bench-uses-vertical-stage-sections-with-natural-scroll.md) for the layout decision), roman-numeral section ticks, per-stage hue identity, parchment dot-grain background, per-row composition (human-readable display name + dotted strategy ID + 4-segment promotion bar + italic eligibility deck + activity bullet + tabular-numerics row), and within-section drag for priority reorder. The v0.3 prototype ([`docs/prototypes/strategy-kanban-v0.3/`](../prototypes/strategy-kanban-v0.3/)) was the original visual reference and remains valuable historical context for editorial-press direction and per-stage hue identity, but the layout contract is now vertical stage sections, not five columns.

2. **The within-stage drag mechanic is variant B with the sticky 50% rule — for priority reorder only.** Locked by the drag-vibes prototype. This is within-section priority reorder; it does not apply to cross-stage transitions. The ghost-landing pattern (destination row holds `visibility: hidden` until ghost completes its glide animation) is part of the locked visual spec and applies to within-section reorder.

3. **Editorial-press direction extends to the Bench surface.** Per-stage hue identity (idle parchment / backtest slate / paper sage / micro amber / live parchment-bright), parchment dot-grain background overlay (~0.025 opacity, 3px spacing), roman-numeral section markers, italic Newsreader section decks, masthead with `session armed | N configs | M live | DATE` meta. These extensions to [DESIGN_SYSTEM.md](../DESIGN_SYSTEM.md) are accepted in principle and will be amended into the design system when Phase 6 work begins. (Note: per-stage hues introduce a *stage* color axis distinct from the *outcome* status colors defined in [DESIGN_SYSTEM.md §6](../DESIGN_SYSTEM.md) and the status-as-nouns principle in [DESIGN.md §5.6](../DESIGN.md). The reconciliation — whether stage hues are tokenized as a parallel namespace or composed from existing tokens — is part of the Phase 6 design-system amendment, not this ADR.)

4. **All architectural mechanics decisions were explicitly deferred to follow-up ADRs.** The list below names the questions; this ADR does not answer them. ADRs 0039-0048 now answer the set and make Phase 6 implementation planning concrete.

5. **The Bench surface is Phase 6+ implementation scope.** It was not authorized for Phase 5. Phase 5 close-out was unaffected by this ADR's existence. This ADR records intent; it did not extend Phase 5.

## Locked visual spec — what the prototype establishes

These are the elements the bench mockup establishes and this ADR accepts as the visual contract for any future implementation:

- **Surface layout.** Five vertical stage sections top-to-bottom (defer to [ADR 0048](0048-bench-uses-vertical-stage-sections-with-natural-scroll.md)): `IDLE → BACKTEST → PAPER → MICRO LIVE → LIVE`. Each section is a stage in the promotion pipeline. IDLE is the precondition (configured, not yet evaluated); the four others are the production stages.
- **Promotion bar.** Per-row 4-segment visualization (BT / PA / ML / LV) showing where each strategy currently sits on the ladder. IDLE strategies render an empty/dashed bar — `PROMO_STAGES = STAGES.slice(1)` in the prototype's data layer; IDLE is *not* on the bar.
- **Per-stage hue.** Each section carries a hue identity (idle parchment, backtest slate, paper sage, micro amber, live parchment-bright). The hue propagates to the section header, the stage chip color, the promotion-bar segment fill, and the masthead's `M live` indicator.
- **Row composition.** Each row carries: human-readable display name (`"ATR Channel Breakout"`) + dotted strategy ID (`breakout.daily.atr_channel.sector_etfs.v1`) + stage chip (top-right) + 4-segment promotion bar + italic eligibility deck (`"gates passing"`, `"max-dd over 15% — needs reparam"`, `"locked by ADR 0004"`, etc.) + activity bullet (`"paper · 3 open · day 88"`) + tabular-numerics row (Sharpe / max-DD / trade count).
- **Drag verbs.** Within-section drag = priority reorder only (variant B + sticky 50% rule from drag-vibes). Cross-stage state changes are not gestures; see [ADR 0047](0047-bench-action-availability-is-the-validation-surface.md) — they are Action menu items computed from gate evidence.
- **Editorial chrome.** Roman-numeral section ticks (`i. ii. iii. iv. v.`), italic Newsreader section decks under each section heading, parchment dot-grain background overlay, masthead meta line. The bench mockup's exact dimensions, opacities, and typography decisions are the spec.
- **Ghost-landing pattern.** When a row is dropped within a section, the data layer commits to the new slot but the destination row renders `visibility: hidden` until the dragged ghost completes its glide animation (~290ms). Once the ghost arrives, the destination becomes visible. Prevents the double-render artifact present in v0.2. Applies to within-section reorder only (no cross-section drop exists). This pattern transfers to QML as: a `Behavior` on opacity gated by a `landing` property the drop handler sets, with a timed reset matching the drop-animation duration.
- **Activity-copy voice.** The prototype's `eligibleNote` strings set the operator-product writing register: concise, evidence-first, and unsentimental. Future implementations preserve that voice while replacing demo-only live countdown or live-readiness copy with policy-accurate locked/evidence copy.

## Deferred mechanics decisions — to be answered before Phase 6 implementation

Each of the following was a real architectural decision the v0.3 prototype hand-waved with demo data. This ADR originally deferred them; ADRs 0039-0048 now answer the set and make Phase 6 implementation planning concrete.

- **Q-A — Display-name provenance.** Answered by [ADR 0041](0041-bench-display-names-are-presentation-metadata.md): optional `strategy.display_name`, presentation-only, with deterministic `strategy_id` fallback.
- **Q-B — Stage / session decoupling.** Answered by [ADR 0039](0039-stage-session-and-bench-section-are-distinct.md): promotion stage, runtime session state, and Bench section are distinct axes.
- **Q-C — Live-eligibility-window rule.** Answered by [ADR 0042](0042-live-and-micro-live-eligibility-is-locked-and-evidence-based.md): ADR 0004 remains authoritative, calendar countdowns are rejected, and eligibility copy stays evidence-based.
- **Q-D — Demotion-action security gradient.** Answered by [ADR 0043](0043-bench-demotion-actions-open-a-governance-flow.md): a demotion Action-menu item opens a governance flow; selection alone never mutates state.
- **Q-E — Action-availability surface.** Answered by [ADR 0047](0047-bench-action-availability-is-the-validation-surface.md): action availability is computed pre-menu from cached verdicts; unavailable actions are hidden, not disabled.
- **Q-F — Responsive layout strategy.** Answered by [ADR 0048](0048-bench-uses-vertical-stage-sections-with-natural-scroll.md): vertical stage sections with native scroll; no horizontal board.
- **Q-G — Bulk-action wiring.** Answered by [ADR 0040](0040-bench-bulk-orchestration-uses-a-durable-job-ledger.md): parent/child orchestration jobs link to concrete execution records.
- **Q-H — Token reconciliation.** Answered by [ADR 0046](0046-bench-stage-hues-extend-production-tokens.md): production tokens remain canonical and stage hues become a separate namespace. Prototype palette drift noted:
  - `--brand-accent` is `#863a44` (prototype) vs `#722f37` (production Editorial Dark)
  - `--text-primary` is `#e4d2b0` (prototype) vs `#d8c5a3` (production Editorial Dark)
  - The prototype defines stage hues (`--stage-idle`, `--stage-backtest`, etc.) that don't exist in production yet.
  - Decision needed: which palette is canonical? Either update production tokens to match the brighter prototype, or update the prototype's tokens (in any future amendment) to match production. The stage-hue tokens need to land in `Theme.qml`.

## Rationale

**Visual-locked-before-mechanics-defined is the right ADR shape for this artifact.** ADRs work best when they lock one thing decisively and explicitly defer everything else. The operator's framing — "love the way it feels, looks, and is laid out; mechanics don't have a whole lot of thought" — splits the artifact cleanly. Trying to lock both layers in one ADR would either (a) defer the visual decision while waiting for mechanics, losing the operator's fresh visual judgment, or (b) commit to mechanics decisions that haven't been thought through. Splitting them keeps the visual contract sharp and the mechanics open.

**The visual contract preserves the editorial-press direction at scale.** Phase 5 established editorial press as the design direction (Newsreader serif, Public Sans body, JetBrains Mono on tabular data, three themes, "the deserving list" / "the failures" voice). The Anchor view and Strategy Bank surface land that direction at *single-surface* scale. The v0.3 prototype lands it at *multi-section, multi-card, full-screen-application* scale — proving the direction holds when the surface gets bigger and busier. Without an ADR locking that visual contract, future Phase 6 work could easily drift toward generic dashboard-toolkit conventions (gradients, glassmorphism, shadow-heavy cards, gradient-text) that erode the editorial print identity.

**The deferred mechanics list prevented Phase 6 from starting blind.** Phase 6 opened with the Bench surface as a likely anchor program. Capturing Q-A through Q-H gave the follow-up ADRs a clear register instead of letting those questions surface one bug at a time.

**The drag-mechanic decisions transfer cleanly into QML.** Variant B + sticky 50% rule (drag-vibes prototype) and the ghost-landing pattern (bench mockup) are both Pointer-Events-API patterns in the prototype. Both port to Qt's `Drag` attached property + `MouseArea preventStealing` + `Behavior on opacity` with comparable semantics for within-section reorder. The visual contract therefore implies the QML mechanism without committing to it; the QML port is implementation, not a re-design.

**Phase 5 close-out was unblocked.** Phase 5's exit criteria were observability-first GUI shipped + installer ships ([ADR 0034](0034-phase-5-scope-orders-observability-before-features.md)). Neither required the Kanban surface. Capturing the Kanban as a forward-looking ADR kept it out of Phase 5 scope, and ADR 0038 later closed the phase.

## Consequences

- **The bench mockup is the canonical visual reference.** Future Phase 6 implementation PRs cite this ADR and the mockup files at [`docs/mockups/bench-brief.md`](../mockups/bench-brief.md) and [`docs/mockups/bench-surface.html`](../mockups/bench-surface.html). The v0.3 prototype ([`docs/prototypes/strategy-kanban-v0.3/`](../prototypes/strategy-kanban-v0.3/)) remains valuable as historical context for editorial-press direction and per-stage hue identity; it is gitignored (per `docs/prototypes/` directory convention) and reproducible via its README at [`docs/prototypes/strategy-kanban-v0.3/README.md`](../prototypes/strategy-kanban-v0.3/README.md) plus this ADR's locked visual-spec section.
- **The Bench is Phase 6+ scope.** Not authorized for Phase 5. Any Phase 5 PR that touches the Bench surface or its underlying mechanics is out-of-scope and would require either rescoping Phase 5 (a fresh ADR) or rolling the work into Phase 6 explicitly.
- **Phase 6 implementation is gated on the mechanics ADRs.** Q-A through Q-H are now answered by ADRs 0039-0048. The first implementation PR should still be read-only foundation work before Action-menu-driven promotions, demotion submissions, or bulk execution.
- **DESIGN_SYSTEM.md gets a Phase 6 amendment** that adopts per-stage hue identity tokens (`stage.idle`, `stage.backtest`, etc.) and QML-accessible texture support. [ADR 0046](0046-bench-stage-hues-extend-production-tokens.md) keeps production tokens canonical and separates stage hue from outcome status color.
- **The drag-vibes prototype mechanics port forward.** Variant B + sticky 50% rule for within-section priority reorder; the ghost-landing pattern for within-section drop. Both move into QML when the Bench surface implementation begins. The drag-vibes prototype itself can be archived alongside the v0.3 prototype as historical reference.
- **The current Strategy Bank surface remains.** It is a *read-only* observability surface for paper-stage and blocked-at-backtest strategies. The Bench supersedes it as a *write-capable* surface (Action-menu-driven promotions, demotion submissions, run/session initiation) but does not replace it on day one of Phase 6. Both can coexist; the Strategy Bank stays as the at-a-glance current-state inspector while the Bench is for active operations.
- **`Theme.column.chips` and the column-reservation contract** ([DESIGN_SYSTEM.md §4.1](../DESIGN_SYSTEM.md#41-column-widths)) extend to Bench row layouts. Rows reserve fixed-width slots for chips, decks, and metric rows; content does not determine column width. The same rule that prevents BLOCKED-section drift on Strategy Bank applies to Bench rows (per [ADR 0048](0048-bench-uses-vertical-stage-sections-with-natural-scroll.md) Decision 4).
- **No live-trading boundary is moved.** [ADR 0004](0004-paper-only-phase-one.md) and [ADR 0005](0005-kill-switch-manual-reset.md) remain authoritative. Promoting to live or starting a live/micro-live session via the Bench Action menu is not actionable unless a future ADR explicitly supersedes ADR 0004 and preserves the human-review gates. The Bench surface is a UX over existing rules, not a relaxation of them.
- **Follow-up ADRs now answer the deferred Qs.** The Phase 6 decision set is [ADR 0039](0039-stage-session-and-bench-section-are-distinct.md) through [ADR 0048](0048-bench-uses-vertical-stage-sections-with-natural-scroll.md).
