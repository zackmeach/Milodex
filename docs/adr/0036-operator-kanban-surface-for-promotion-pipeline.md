# ADR 0036 — Operator Kanban surface for the promotion pipeline

**Status:** Visual spec accepted; mechanics deferred · 2026-05-08
**Related:** [ADR 0033](0033-gui-runtime-is-pyside6-qt-quick.md) (GUI runtime), [ADR 0034](0034-phase-5-scope-orders-observability-before-features.md) (Phase 5 observability-first), [ADR 0035](0035-design-system-and-theme-architecture.md) (design system architecture), [ADR 0005](0005-kill-switch-manual-reset.md) (manual-reset semantic, applies to drag-to-demote), [PHASE5_PLANNING.md](../PHASE5_PLANNING.md), [STRATEGY_BANK.md](../STRATEGY_BANK.md), [DESIGN_SYSTEM.md](../DESIGN_SYSTEM.md), [`docs/prototypes/strategy-kanban-v0.3/`](../prototypes/strategy-kanban-v0.3/) (canonical visual spec)

## Context

[ADR 0034](0034-phase-5-scope-orders-observability-before-features.md) bounded Phase 5 to observability surfaces. By the end of Phase 5 the Anchor view (operator first-frame, kill-switch + market clock + account + positions) and the Strategy Bank surface (read-only render of paper-stage and blocked-at-backtest strategies) had shipped, plus the design-system foundation (tokens, three themes, components, editorial-press direction) and the QML-load smoke test. Phase 5 close-out remains: distribution-model ADR, installer PR, and a Phase-5-closes ADR. No feature surface — anything that gives the operator new strategy operations beyond the existing CLI — is in Phase 5 scope.

Late in Phase 5 the operator surfaced an interaction-language idea: drag-to-reorder rows in the Strategy Bank surface, possibly serializing deployment as a queue. Investigation (the drag-vibes prototype at [`docs/prototypes/strategy-bank-drag-vibes.html`](../prototypes/strategy-bank-drag-vibes.html), three iterations of variant trade-offs) validated a specific drag mechanic — variant B with the sticky 50% rule — that feels right.

The conversation then expanded substantially. The operator described a full Kanban-style operator surface for the promotion pipeline:

> "I would like in front of me to be every single strategy that I have access to. I would like to be able to start with all 12 or 13 runners or configs in the pre-backtest or untested category. Drag a few that I want into the backtest section. Hit start, which would then run backtesting on all the strategies in that section. Depending on the results, then update the small chip that's visible for that each strategy that shows whether this is backtested able to promote or backtested not able to be promoted. Then I can drag the few that are able to be backtested that have passed backtesting over to paper testing. I could even try and drag some of the items that are backtested not ready to promote into the paper testing, and then would be shown a denied message…There should also be on that strategies row a running or idle chip…"

The operator then iterated visually on the idea, producing a self-contained React prototype (v0.2 then v0.3, latest at [`docs/prototypes/strategy-kanban-v0.3/`](../prototypes/strategy-kanban-v0.3/)) that spans five columns (IDLE / BACKTEST / PAPER / MICRO LIVE / LIVE), full editorial-press treatment (roman-numeral column ticks, parchment dot-grain background, per-stage hue identity, italic Newsreader decks throughout), per-card promotion bar visualization, bulk-action buttons with smart-skip ×N counters, and the drag mechanic from the earlier prototype extended to cross-column promotions and demotions.

The operator's framing of the prototype, verbatim:

> "the mechanics of what is going on, how strategies are run, how the content in each card gets updated, doesn't have a whole lot of thought…I love the way it feels, the way it looks, and how it is laid out."

That framing splits the artifact cleanly into a layer that is locked (visual spec, layout, UX direction) and a layer that is not yet defined (mechanics, data model, runtime). This ADR adopts the visual spec and explicitly defers the mechanics layer to follow-up ADRs. The intent is to capture the surface's shape now — while the operator's thinking is fresh, before Phase 5 closes out — without committing to mechanics decisions that haven't been thought through yet.

The Kanban surface is **Phase 6+ implementation work**. Phase 5 close-out is not blocked on it. This ADR is a forward-looking design freeze, not a Phase 5 deliverable.

## Decision

1. **Adopt the v0.3 prototype as the canonical visual spec for the operator Kanban surface.** [`docs/prototypes/strategy-kanban-v0.3/`](../prototypes/strategy-kanban-v0.3/) is the reference artifact. Future Phase 6 implementation PRs treat this prototype as the visual contract: five columns, roman-numeral ticks, per-stage hue identity, dot-grain background, per-card composition (human-readable name + dotted strategy ID + 4-segment promotion bar + italic eligibility deck + activity bullet + tabular-numerics row), bulk-action `×N` buttons, drag-to-reorder within columns, drag-to-promote/demote across columns, and the ghost-landing pattern that prevents the v0.2 double-render artifact.

2. **The within-column drag mechanic is variant B with the sticky 50% rule.** Locked by the drag-vibes prototype. The ghost-landing pattern (destination card holds `visibility: hidden` until ghost completes its glide animation) is part of the locked visual spec.

3. **Editorial-press direction extends to the Kanban surface.** Per-stage hue identity (idle parchment / backtest slate / paper sage / micro amber / live parchment-bright), parchment dot-grain background overlay (~0.025 opacity, 3px spacing), roman-numeral column markers, italic Newsreader column decks, masthead with `session armed | N configs | M live | DATE` meta. These extensions to [DESIGN_SYSTEM.md](../DESIGN_SYSTEM.md) are accepted in principle and will be amended into the design system when Phase 6 work begins.

4. **All architectural mechanics decisions are explicitly deferred to follow-up ADRs.** The list below names the questions; this ADR does not answer them. Each becomes its own ADR (or grouped scope decision) when Phase 6 opens. Phase 6 implementation cannot ship until the relevant follow-up ADRs land.

5. **The Kanban surface is Phase 6+ implementation scope.** It is not authorized for Phase 5. Phase 5 close-out (distribution model + installer + close ADR) is unaffected by this ADR's existence. This ADR records intent; it does not extend Phase 5.

## Locked visual spec — what the prototype establishes

These are the elements the v0.3 prototype demonstrates and this ADR accepts as the visual contract for any future implementation:

- **Surface layout.** Five columns left-to-right: `IDLE → BACKTEST → PAPER → MICRO LIVE → LIVE`. Each column is a stage in the promotion pipeline. IDLE is the precondition (configured, not yet evaluated); the four others are the production stages.
- **Promotion bar.** Per-card 4-segment visualization (BT / PA / ML / LV) showing where each strategy currently sits on the ladder. IDLE strategies render an empty/dashed bar — `PROMO_STAGES = STAGES.slice(1)` in the prototype's data layer; IDLE is *not* on the bar.
- **Per-stage hue.** Each column carries a hue identity (idle parchment, backtest slate, paper sage, micro amber, live parchment-bright). The hue propagates to the column's chip color, the promotion-bar segment fill, and the masthead's `M live` indicator.
- **Card composition.** Each card carries: human-readable display name (`"ATR Channel Breakout"`) + dotted strategy ID (`breakout.daily.atr_channel.sector_etfs.v1`) + stage chip (top-right) + 4-segment promotion bar + italic eligibility deck (`"gates passing — micro-live ready"`, `"max-dd over 15% — needs reparam"`, `"21 days to live-eligibility window"`, etc.) + activity bullet (`"paper · 3 open · day 88"`) + tabular-numerics row (Sharpe / max-DD / trade count).
- **Bulk-action button per non-IDLE column.** `RUN ×N` for backtest, `SESSION ×N` for paper / micro-live. The `×N` counter shows how many strategies in that column are awaiting action — the smart-skip ergonomic. LIVE has no bulk-action button; live is a state, not a verb.
- **Drag verbs.** Within-column drag = priority reorder (variant B + sticky 50% rule from drag-vibes). Cross-column drag = stage transition request, runs validation against `state_machine.py` rules. Reverse cross-column drag = demotion (security-gated, see deferred Q-D below).
- **Editorial chrome.** Roman-numeral column ticks (`i. ii. iii. iv. v.`), italic Newsreader column decks under each section heading, parchment dot-grain background overlay, masthead meta line. The v0.3 prototype's exact dimensions, opacities, and typography decisions are the spec.
- **Ghost-landing pattern.** When a card is dropped, the data layer commits to the new slot but the destination card renders `visibility: hidden` until the dragged ghost completes its glide animation (~290ms). Once the ghost arrives, the destination becomes visible. Prevents the double-render artifact present in v0.2. This pattern transfers to QML as: a `Behavior` on opacity gated by a `landing` property the drop handler sets, with a timed reset matching the drop-animation duration.
- **Activity-copy voice.** The prototype's `eligibleNote` strings ("config valid — ready to backtest", "walk-forward complete — paper-ready", "max-dd over 15% — needs reparam", "sharpe holds — promote to live?", "21 days to live-eligibility window") set the operator-product writing register. Future implementations preserve this voice.

## Deferred mechanics decisions — to be answered before Phase 6 implementation

Each of the following is a real architectural decision the v0.3 prototype hand-waves with demo data. None is answered by this ADR; each becomes a follow-up ADR or design task at Phase 6 open.

- **Q-A — Display-name provenance.** The prototype's `name` field (`"ATR Channel Breakout"`) does not exist in production configs. Decision needed: new YAML field on each strategy config? Derivation rule from `strategy_id`? Hand-curated table in a metadata file? Each option has different schema-migration consequences.
- **Q-B — Stage / session decoupling.** The prototype implicitly distinguishes "stage" (where on the ladder) from "session" (whether a runner process is alive) via a `paused: true` flag with ambiguous semantics. Production today conflates the two — a strategy "is at paper" if a paper-stage promotion record exists, period. Decision needed: separate session state into `strategy_runs` semantics, a new sessions table, or process-level reconciliation. Affects the runner's startup/shutdown contract and how the GUI queries "what's running right now."
- **Q-C — Live-eligibility-window rule.** The prototype's "21 days to live-eligibility window" deck implies a minimum-time-at-micro-before-live rule. No such rule exists in `state_machine.py`. Decision needed: is the gate time-based (days at micro), trade-count-based (N trades observed at micro), Sharpe-stability-based (rolling-window Sharpe holds above threshold), or some combination. ADR-sized in itself; touches gating policy.
- **Q-D — Demotion-gesture security gradient.** Drag-to-promote (e.g. PAPER → MICRO LIVE) is one gesture. Drag-to-demote (e.g. MICRO LIVE → PAPER, which kills capital and stops a runner) is conceptually the same gesture but operationally far more consequential. [ADR 0005](0005-kill-switch-manual-reset.md) and [CLAUDE.md](../../CLAUDE.md)'s autonomy-boundary list both demand explicit human approval for actions of this consequence. Decision needed: does demotion require typed-confirmation (kill-switch reset pattern), longer hold-time, two-step gesture, or something else. The drag mechanic itself must encode this gradient.
- **Q-E — Hover-validation timing.** When the user drags a forbidden card over a target column (e.g. an unbacktested strategy dragged to PAPER, which `state_machine.py` would reject), does the column show denial *during hover* (column tints rust, denied tooltip appears) or only *on drop*? Hover-validation gives instant feedback at the cost of running the validator on every drag-over event. Drop-validation is simpler. UX gap in the prototype either way.
- **Q-F — Responsive layout strategy.** The prototype is 5 columns × ~280px = ~1400px content width. Comfortable at 1920px+. At 1280px-laptop widths the columns overflow. Decision needed: column-wrap, horizontal scroll, column-collapse-with-tabs, or some hybrid. Affects the Qt layout primitives chosen for the QML port.
- **Q-G — Bulk-action wiring.** "Start" on the BACKTEST column dispatches N backtests for queued strategies. Production has `walk_forward_batch.py` which already orchestrates batched walk-forward runs — does the bulk action map onto that, or does it need a new orchestrator that handles per-strategy progress reporting back to the GUI? Subprocess-per-strategy or shared-process? How does cancellation work? How is per-strategy progress (`bt 74% · 2014-2024` in the prototype) surfaced from the runner back to the card?
- **Q-H — Token reconciliation.** Prototype palette drifts slightly from production:
  - `--brand-accent` is `#863a44` (prototype) vs `#722f37` (production Editorial Dark)
  - `--text-primary` is `#e4d2b0` (prototype) vs `#d8c5a3` (production Editorial Dark)
  - The prototype defines stage hues (`--stage-idle`, `--stage-backtest`, etc.) that don't exist in production yet.
  - Decision needed: which palette is canonical? Either update production tokens to match the brighter prototype, or update the prototype's tokens (in any future amendment) to match production. The stage-hue tokens need to land in `Theme.qml`.

## Rationale

**Visual-locked-before-mechanics-defined is the right ADR shape for this artifact.** ADRs work best when they lock one thing decisively and explicitly defer everything else. The operator's framing — "love the way it feels, looks, and is laid out; mechanics don't have a whole lot of thought" — splits the artifact cleanly. Trying to lock both layers in one ADR would either (a) defer the visual decision while waiting for mechanics, losing the operator's fresh visual judgment, or (b) commit to mechanics decisions that haven't been thought through. Splitting them keeps the visual contract sharp and the mechanics open.

**The visual contract preserves the editorial-press direction at scale.** Phase 5 established editorial press as the design direction (Newsreader serif, Public Sans body, JetBrains Mono on tabular data, three themes, "the deserving list" / "the failures" voice). The Anchor view and Strategy Bank surface land that direction at *single-surface* scale. The v0.3 prototype lands it at *multi-section, multi-card, full-screen-application* scale — proving the direction holds when the surface gets bigger and busier. Without an ADR locking that visual contract, future Phase 6 work could easily drift toward generic dashboard-toolkit conventions (gradients, glassmorphism, shadow-heavy cards, gradient-text) that erode the editorial print identity.

**The deferred mechanics list prevents Phase 6 from starting blind.** Phase 6 will open with the Kanban surface as a likely anchor program. Without this ADR, Phase 6's first weeks would be spent rediscovering the questions Q-A through Q-H one bug at a time. Capturing them now as deferred-but-named gives Phase 6 a clear opening register: each question becomes a small follow-up ADR or design task, addressed before the relevant feature lands.

**The drag-mechanic decisions transfer cleanly into QML.** Variant B + sticky 50% rule (drag-vibes prototype) and the ghost-landing pattern (Kanban v0.3 prototype) are both Pointer-Events-API patterns in the prototype. Both port to Qt's `Drag` attached property + `MouseArea preventStealing` + `Behavior on opacity` with comparable semantics. The visual contract therefore implies the QML mechanism without committing to it; the QML port is implementation, not a re-design.

**Phase 5 close-out is unblocked.** Phase 5's exit criteria are observability-first GUI shipped + installer ships ([ADR 0034](0034-phase-5-scope-orders-observability-before-features.md)). Neither requires the Kanban surface. Capturing the Kanban as a forward-looking ADR clears it off the open-questions register without bringing it into Phase 5 scope. The distribution-model ADR, installer PR, and Phase-5-closes ADR proceed unaffected.

## Consequences

- **The v0.3 prototype is the canonical visual reference.** Future Phase 6 implementation PRs cite this ADR and the prototype directory. The prototype itself is gitignored (per `docs/prototypes/` directory convention) — it is reference material the operator opens locally, not a source artifact in source control. Reproducibility is via the README at [`docs/prototypes/strategy-kanban-v0.3/README.md`](../prototypes/strategy-kanban-v0.3/README.md) plus this ADR's locked visual-spec section.
- **The Kanban is Phase 6+ scope.** Not authorized for Phase 5. Any Phase 5 PR that touches the Kanban surface or its underlying mechanics is out-of-scope and would require either rescoping Phase 5 (a fresh ADR) or rolling the work into Phase 6 explicitly.
- **Phase 6 implementation is gated on the deferred mechanics ADRs.** Q-A through Q-H must be answered (each via an appropriate small ADR or design task) before the corresponding Kanban feature ships. The operator can still scope Phase 6 to ship a partial Kanban surface (e.g. read-only view across stages without drag-to-promote) before all deferred ADRs land — but feature-by-feature, the gating questions need answers.
- **DESIGN_SYSTEM.md gets a Phase 6 amendment** that adopts: per-stage hue identity tokens (`color.stage.idle`, `color.stage.backtest`, etc.), the parchment-dot-grain background pattern, the masthead-meta line specification, the roman-numeral column-tick convention, and the ghost-landing visibility-hidden pattern. These extensions wait until Phase 6 opens and Q-H (token reconciliation) is decided.
- **The drag-vibes prototype mechanics port forward.** Variant B + sticky 50% rule for within-column reorder; the ghost-landing pattern for cross-column drop. Both move into QML when the Kanban surface implementation begins. The drag-vibes prototype itself can be archived alongside the v0.3 prototype as historical reference.
- **The current Strategy Bank surface remains.** It is a *read-only* observability surface for paper-stage and blocked-at-backtest strategies. The Kanban supersedes it as a *write-capable* surface (drag-to-promote, bulk run/session) but does not replace it on day one of Phase 6. Both can coexist; the Strategy Bank stays as the at-a-glance current-state inspector while the Kanban is for active operations.
- **`Theme.column.chips` and the column-reservation contract** ([DESIGN_SYSTEM.md §4.1](../DESIGN_SYSTEM.md#41-column-widths)) extend to Kanban card layouts. Cards reserve fixed-width slots for chips, decks, and metric rows; content does not determine column width. The same rule that prevents BLOCKED-section drift on Strategy Bank applies to Kanban cards.
- **No live-trading boundary is moved.** [ADR 0004](0004-paper-only-phase-one.md) and [ADR 0005](0005-kill-switch-manual-reset.md) remain authoritative. Drag-to-promote-to-live, when implemented, fires a typed-confirmation dialog (kill-switch-reset pattern); drag-to-demote-from-live similarly. The Kanban surface is a UX over existing rules, not a relaxation of them. Q-D captures the security-gradient design decision.
- **A future ADR addresses each deferred Q.** When Phase 6 work begins on a feature gated by one of Q-A through Q-H, the corresponding ADR is drafted and accepted before the implementation PR lands. The follow-up ADR numbering picks up at 0037+.
