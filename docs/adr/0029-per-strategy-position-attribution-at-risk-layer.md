# ADR 0029 — Per-Strategy Position Attribution at the Risk Layer

**Status:** Accepted · 2026-05-06
**Implementation update (2026-05):** Backtests now expose explicit risk-policy modes. The default `RiskPolicy.BYPASS` mode preserves this ADR's original raw-research behavior with `NullRiskEvaluator`; opt-in `RiskPolicy.ENFORCE` applies backtest structural sizing and exposure checks.
**Related:** [ADR 0024](0024-account-scoped-position-caps-are-authoritative.md) (extended, not superseded), [ADR 0028](0028-phase-4-scope-closes-as-cleanup-and-attribution.md) (authorizes this work), [PHASE2_PLANNING.md §3.2 CS-1](../PHASE2_PLANNING.md), [ADR 0022](0022-strategy-rotation-scope-is-the-declared-universe.md) (intent-side discipline that complements this execution-side discipline), [ADR 0026](0026-concurrent-multi-strategy-uses-per-process-supervisor.md) (concurrency model this enables), [ADR 0011](0011-sqlite-event-store.md) (the source of truth for attribution data), [ADR 0008](0008-risk-layer-veto-architecture.md) ("strategy proposes, risk disposes"), [CLAUDE.md](../../CLAUDE.md) "Risk layer is sacred"

## Context

[ADR 0024](0024-account-scoped-position-caps-are-authoritative.md) named the schema overload behind the 2026-05-04 regime/meanrev incident: strategy YAML's `risk.max_positions` reads naturally as a per-strategy ceiling, but the risk evaluator's `concurrent_positions` check counts every open broker position regardless of which strategy proposed it. ADR 0024 chose option (c) — document the overload, accept account-scoped enforcement as authoritative, treat strategy YAML's `risk.max_positions` as informational metadata. That resolution preserved [ADR 0024](0024-account-scoped-position-caps-are-authoritative.md)'s Consequences clause: *"if/when the operator opens that goal, option (a) becomes the right next move and this ADR's semantics extend rather than reverse."*

[ADR 0028](0028-phase-4-scope-closes-as-cleanup-and-attribution.md) opens that goal. Phase 4 §4.1 = (f) authorizes per-strategy position attribution at the risk layer; the operator's mechanics-firming intent ([ADR 0028 Rationale](0028-phase-4-scope-closes-as-cleanup-and-attribution.md)) names the concrete operational pull: *"I can't quickly tell what strategies exist, where they are, how much they've made."* Per-strategy attribution at the risk layer is the prerequisite for any future surface — CLI report, GUI dashboard, P&L breakdown — to disambiguate strategy ownership of broker positions truthfully.

The architectural extension is small in surface area but precise in semantics. Three constraints scope it:

1. **The broker is the position arbiter.** Alpaca's `Position` carries no strategy attribution ([src/milodex/broker/models.py:67](../../src/milodex/broker/models.py)). Per [ADR 0010](0010-hybrid-source-of-truth.md) ("hybrid source of truth"), live position state is queried from the broker per evaluation, not stored in the event store. Attribution must be reconstructed from durable history on demand.

2. **The event store already records the attribution chain.** Per [ADR 0011](0011-sqlite-event-store.md), the `trades` table carries `strategy_name` (= the runner's `strategy_id` for runner-submitted trades, `NULL` for operator trades), `submitted_by`, `symbol`, `side`, `quantity`, `status`, `broker_order_id`, and `recorded_at`. The opening fill of a current position is recoverable by walking that history — but only the rows where the trade was actually submitted to the broker count as fills (see Decision 2).

3. **ADR 0024's account-scoped floor stays authoritative.** [CLAUDE.md](../../CLAUDE.md)'s "risk layer is sacred" — and [FOUNDER_INTENT.md](../FOUNDER_INTENT.md) priority #1 (trustworthy) — require that the account never exceed its global cap. Per-strategy enforcement is **additive**, not substitutive. Both checks must pass.

This ADR articulates the eight architectural decisions PR #39 will implement.

## Decision

1. **Position ownership is anchored to the opening fill.** A broker position belongs to the strategy whose runner submitted its opening fill (the BUY that took the symbol from zero shares held to a non-zero balance). Subsequent increases preserve attribution. Partial liquidations leave attribution unchanged on the remaining shares. Full liquidation followed by a fresh opening BUY creates a new attribution.

2. **Attribution is reconstructed from `trades` on demand — no new table.** When the risk evaluator needs to know "how many positions does strategy X currently own?", it queries the broker for current positions (the authoritative live state per [ADR 0010](0010-hybrid-source-of-truth.md)) and, for each non-zero-quantity symbol, walks the symbol-indexed `trades` rows backwards to locate the most recent zero → non-zero opening fill. **Only rows where the trade was actually submitted to the broker count as fills** — concretely, rows with `status="submitted"` (equivalently, `broker_order_id IS NOT NULL`). Rows with `status` in `{"preview", "blocked", "cancelled"}` MUST be excluded from the walk: a `side="buy"` row with `status="blocked"` is a rejected intent, not an opening fill, and counting it would misattribute the symbol to whichever strategy proposed the rejected trade. The `strategy_name` column on the surviving (submitted) opening trade is the symbol's attribution. No `position_attribution` table; no schema migration; no on-fill bookkeeping that could drift from the broker's authoritative view.

3. **Pre-attribution positions are operator-attributed.** A current broker position whose opening fill (resolved per Decision 2) was placed by the operator — i.e. the row's `strategy_name IS NULL` (which co-occurs with `submitted_by="operator"` on operator-submitted trades) — or whose symbol has no recoverable submitted opening fill in the event store at all, is treated as belonging to the special pseudo-strategy `"operator"`. The reliable attribution signal is `strategy_name`: it is `NULL` exactly when the operator submitted the trade (because `_build_execution_request` ([src/milodex/execution/service.py:402](../../src/milodex/execution/service.py)) sets `strategy_name=strategy_config.name if strategy_config else None`, and operator manual trades have no `strategy_config_path`). `submitted_by` is a co-occurring tertiary signal but is not the primary key — for runner trades it is the literal constant string `"strategy_runner"` ([src/milodex/strategies/runner.py:330](../../src/milodex/strategies/runner.py)), not the runner's `strategy_id`. Operator-attributed positions count toward the account-scoped cap (ADR 0024 unchanged) but **do not** count toward any runner-strategy's per-strategy cap.

4. **Operator-placed orders are operator-attributed by definition.** `milodex trade submit` already passes `submitted_by="operator"` and a `strategy_config_path=None` on the `TradeIntent` ([src/milodex/execution/models.py:40](../../src/milodex/execution/models.py)). The execution service resolves that to `strategy_name=None` on the recorded `trades` row (via `_build_execution_request` at [src/milodex/execution/service.py:402](../../src/milodex/execution/service.py)). Positions opened by operator submission therefore land in the `"operator"` pseudo-strategy bucket via Decision 3's `strategy_name IS NULL` resolution path — no special case is needed at execution time.

5. **The risk evaluator runs both caps; either failing blocks the trade.** A new `_check_strategy_concurrent_positions` runs alongside the existing `_check_concurrent_positions` (which is unchanged). Both must pass. The new check produces reason code `max_strategy_positions_exceeded` distinct from the existing `max_concurrent_positions_exceeded`. When both caps are exceeded by the same intent, both reason codes appear in the explanation record's `reason_codes` array and both checks appear in `risk_checks` — the existing infrastructure ([src/milodex/risk/evaluator.py:93-95](../../src/milodex/risk/evaluator.py)) already supports multiple-failure decisions.

6. **Strategy YAML `risk.max_positions` becomes binding for the per-strategy cap.** Its interpretation is now: *"this strategy can never own more than N broker positions attributed to it, regardless of account-scoped headroom."* `risk_defaults.yaml`'s `max_concurrent_positions` remains the account-wide floor — unchanged. A strategy YAML with no `risk.max_positions` set: the per-strategy check is **skipped** (treated as no per-strategy limit). The account-scoped floor still applies.

7. **ADR 0024 is extended, not superseded.** The account-scoped check (`_check_concurrent_positions`) remains in place with identical semantics. Tests pinning the existing behavior — notably `tests/milodex/risk/test_evaluator.py` cases asserting `max_concurrent_positions_exceeded` against the global cap — must remain green after PR #39. The new per-strategy check sits beside the account-scoped check, not in front of it or in place of it.

8. **No schema migration; no feature flag.** Decision 2 leaves the `trades` table unchanged; PR #39 adds no migration. The mechanism activates for any strategy whose YAML carries `risk.max_positions`; it is a no-op for any strategy without that field. Operator-attributed positions and strategies without the field continue under the ADR 0024 regime alone.

## Rationale

- **Why both caps, not just per-strategy.** [CLAUDE.md](../../CLAUDE.md) frames the risk layer as the account's last line of defense before the broker. A strategy whose per-strategy cap allows ten positions but whose runner is started against a paper account already holding ten operator-placed positions must still be blocked — the account is the safety boundary the broker actually exposes. Per-strategy enforcement adds expressiveness; it does not replace the floor. Letting a strategy bypass the account cap because it owns "only its share" of positions would re-open exactly the failure mode ADR 0024 closed.

- **Why "operator"-attributed for pre-attribution positions, not "block-until-labeled."** Option (c) from the original framing — block all execution until pre-attribution positions are manually labeled — is the most rigorous but requires operator action against every paper account that pre-dates this ADR. The `"operator"` pseudo-strategy semantic is a strict no-regression: positions that existed before per-strategy attribution are treated as if the operator placed them by hand, which is true in spirit (no runner claimed them) and produces correct behavior under both caps. Future strategies operate freely against the account-scoped headroom these positions consume; they cannot accidentally inherit ownership of a position they did not open. This matches the [ADR 0022](0022-strategy-rotation-scope-is-the-declared-universe.md) discipline on the intent side: a strategy cannot infer ownership from broker presence.

- **Why reconstruct from `trades` (option 2(b)) instead of a `position_attribution` table (option 2(a)).** The `trades` table already carries every field required: `strategy_name` (resolved from the strategy YAML's `name` field via `ExecutionRequest.strategy_name` in [src/milodex/execution/service.py:402](../../src/milodex/execution/service.py) — which is the runner's `strategy_id` for runner-submitted trades and `NULL` for operator-submitted trades), `symbol`, `side`, `quantity`, `status`, and `broker_order_id`. It is append-only by [ADR 0011](0011-sqlite-event-store.md), is symbol-indexed (`idx_trades_symbol`), and is the durable source of truth for fills. Maintaining a parallel `position_attribution` table would introduce a write that can drift from the broker's authoritative position state — exactly the reliability hazard [ADR 0010](0010-hybrid-source-of-truth.md) warns against. The reconstruction cost is bounded: the broker reports ≤ `max_concurrent_positions` symbols per evaluation (configured small for Phase 4); each symbol's history is a short index-aided scan over submitted-only rows; the work is O(positions × small-constant) per `evaluate()` call. If profiling later shows the scan dominates evaluator cost, ADR 0029 can be revisited without a semantic change — the attribution rule (Decision 1) is independent of the storage mechanism.

- **Why a new reason code instead of overloading `max_concurrent_positions_exceeded`.** Reason codes are the operator-facing forensic surface — they appear in explanation rows, in `milodex report`, and (eventually) in any GUI. Distinguishing the two failures lets the operator see immediately whether the block is the global cap (raise it or liquidate stale operator positions) or the per-strategy cap (the strategy is at its self-declared ceiling). Conflating them would re-create the schema overload [ADR 0024](0024-account-scoped-position-caps-are-authoritative.md) named.

- **Why "no `risk.max_positions` set" means no per-strategy limit, not "default to N."** Defaulting to a global value would either be too tight (blocking legitimate strategies that legitimately want headroom) or too loose (a no-op that pretends to enforce). A skip-when-absent semantic is honest about the strategy author's intent: if you don't declare a cap, you haven't asked for one. The account-scoped floor still applies. This matches [ADR 0003](0003-config-driven-strategies.md)'s discipline ("config defines tuning") — the absence of a tuning value is a meaningful declaration.

## Consequences

- **PR #39 implements:**
  - A new `_check_strategy_concurrent_positions` method on `RiskEvaluator` ([src/milodex/risk/evaluator.py](../../src/milodex/risk/evaluator.py)) wired into `evaluate()`'s checks list.
  - An attribution helper (location TBD by PR #39 — likely `milodex.risk.attribution` or a new method on `EventStore`) that walks the symbol-indexed `trades` rows, **filtering to `status="submitted"` rows only** per Decision 2, and returns the attributed `strategy_id` (or `"operator"` when the chain resolves to a `strategy_name IS NULL` row per Decision 3, or the chain is absent).
  - Risk evaluator wiring to consume the helper: for each broker position in `context.positions`, resolve attribution, count those matching the proposing strategy, project the post-trade count for the proposing strategy, compare against `context.expected_max_positions` directly. **The new check MUST read `expected_max_positions` straight off `EvaluationContext` — it must NOT call `_effective_max_positions()` ([src/milodex/risk/evaluator.py:415-423](../../src/milodex/risk/evaluator.py)), which applies a `min(global_default, per_strategy)` clamp. That clamp belongs to the existing account-scoped check; the per-strategy cap is now an independent ceiling alongside the global one, not a value derived from it.** When `expected_max_positions is None` (operator manual trades, or a strategy YAML with no `risk.max_positions` field), the per-strategy check is skipped per Decision 6. The runner-bound envelope already plumbs the value through every cycle of a runner ([src/milodex/risk/evaluator.py:68](../../src/milodex/risk/evaluator.py) `expected_max_positions`).
  - New unit and integration tests pinning the new semantics: per-strategy block, account-scoped block, both-block (both reason codes appear), pre-attribution positions counted to operator, partial-liquidation preserves attribution, full-liquidation-then-rebuy creates new attribution, and **blocked/preview/cancelled rows are ignored by the reconstruction walk** (Decision 2 filter).
- ADR 0024's account-scoped enforcement is **preserved as the floor**. Existing tests pinning `max_concurrent_positions_exceeded` semantics remain green.
- New reason code `max_strategy_positions_exceeded` enters the risk vocabulary. `docs/RISK_POLICY.md` adds it alongside the existing list.
- Strategy YAML's `risk.max_positions` becomes **binding** for runner-attributed positions of that strategy. The comment in `configs/sample_strategy.yaml` is updated to reflect the binding interpretation; ADR 0024's note that the field was advisory is amended in `docs/RISK_POLICY.md` (not in ADR 0024 itself — this ADR is the extension point per [ADR 0028 Non-goals](0028-phase-4-scope-closes-as-cleanup-and-attribution.md)).
- Operator-placed orders work unchanged at the execution surface. They are now categorized as `"operator"`-attributed and counted only against the account-scoped cap.
- The runner-bound risk envelope ([src/milodex/risk/evaluator.py:68](../../src/milodex/risk/evaluator.py) `expected_max_positions`) already plumbs `risk.max_positions` through every cycle of a strategy runner. PR #39 reuses that channel — no new TOCTOU surface is introduced.
- The `"operator"` string is a **reserved pseudo-strategy id** going forward. `configs/` MUST NOT define a strategy named `operator`; PR #39 may add a startup-time check guarding this.
- Phase 4's close-out ADR (PR #10) will reference this ADR as a major architectural artifact — the closing of [ADR 0024](0024-account-scoped-position-caps-are-authoritative.md)'s deferred path (a).

## Non-goals

- This ADR does **not** change account-scoped enforcement. [ADR 0024](0024-account-scoped-position-caps-are-authoritative.md)'s `_check_concurrent_positions` is unchanged.
- This ADR does **not** add new fields to the `trades` table beyond what's already present (`strategy_name`, `submitted_by`, etc.).
- This ADR does **not** introduce strategy-to-strategy position transfers. A position attributed to strategy A stays attributed to strategy A until fully liquidated; a subsequent BUY by strategy B is a new attribution starting from zero shares.
- This ADR does **not** change broker-side semantics. Alpaca still sees positions as account-level. Per-strategy attribution is a Milodex-internal layer over a flat broker view.
- This ADR does **not** pre-commit specific implementation code. PR #39 is the implementation; the helper's location, signature, and exact query shape are PR-#7-owned within the constraints stated in Decisions 1–8.
- This ADR does **not** alter [ADR 0026](0026-concurrent-multi-strategy-uses-per-process-supervisor.md)'s per-process model. Two runners writing to the same `data/milodex.db` continue to read each other's `trades` rows when reconstructing attribution — the durable shared state model is what makes per-process attribution coherent across processes.
- This ADR does **not** change the default backtest path. `RiskPolicy.BYPASS` still uses `NullRiskEvaluator` ([src/milodex/risk/policy.py](../../src/milodex/risk/policy.py)) for raw research simulation; `RiskPolicy.ENFORCE` is a later, explicit backtest structural policy path.
- This ADR does **not** define per-strategy *notional* caps. Decision 6 pins per-strategy enforcement to **position count**, consistent with the `max_positions` field's existing semantic. A separate per-strategy notional cap is a Phase 5+ candidate, not a Phase 4 commitment.

## Open questions surfaced for PR #39 implementation

- **Helper placement and signature.** Whether the attribution lookup lives on `EventStore` (e.g. `event_store.attribute_position(symbol)` returning `str | None`), in a new `milodex.risk.attribution` module, or as a method on `RiskEvaluator` is left to PR #39. The constraint: it must be a single call site that the evaluator invokes per position-symbol per evaluation, so the cost is auditable.
- **Exact `trades` query shape.** Decision 2 fixes the **rule** (most recent zero → non-zero opening fill); the **query** (single SQL with running balance, or per-symbol fetch + Python walk) is PR #39's call. The `idx_trades_symbol` index supports either; profiling under realistic data volume (hundreds to low-thousands of trade rows) will pick the simpler path.
- **Explanation rendering when both caps fail.** The existing `risk_checks` array already records every check independently; both reason codes already appear in `reason_codes`. Whether `milodex report` and any future GUI render the dual failure as one combined message or two stacked messages is a presentation question, not an architecture question, and is owned by the surface that consumes the explanation row.
- **Pre-attribution audit (optional).** Whether PR #39 adds a one-shot `milodex reconcile attributions` command that prints the current attribution map (per current broker position: which strategy_id, or `"operator"`) is PR #39's discretion. Useful for operator confidence at first activation; not architecturally required.
- **Rename of the existing reason code (out of scope).** `max_concurrent_positions_exceeded` is the account-scoped failure code today. Decision 5 keeps that name unchanged for backward compatibility with explanation history. A future ADR may rename it to `max_account_positions_exceeded` for symmetry; this ADR explicitly does not.
